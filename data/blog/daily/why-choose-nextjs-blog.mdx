개발자라면 누구나 블로그가 있다라는 생각하고 있었는데 프론트 개발자로서 포스팅보다 블로그 자체에 집착하게 되어 4번 정도 갈아치웠다. <br/>
(네이버블로그 ⇒ 티스토리  ⇒ gPage(jekyll) ⇒ gatsby ⇒ nextjs)

초반 사용했던 블로그들은 CMS 기반이여 족쇄를 채워 기능이 제한되고 테마를 공부하는 데 시간을 다 허비하였다.

<br/>

최종적으로 NextJS를 왜 선택하였는지 장점을 나열해보고자 한다.

지금의 블로그 선택기준은 다음과 같다. 

- 접근성 (글을 막 써내려갈 수 있는)
- SEO
- 개발자 다운 (?) (커밋, 히스토리 관리)
- 확장성

## 접근성

글을 WYSIWYG으로 작성하면 편하지만 역시 관리하기 쉽고 평소에 자주 접근하는 `md(MarkDown)`을 원하게 되었고 하지만 md로는 부족하기에 `mdx(MarkdDown with JSX)`로 향하게 되었다.

mdx는 Component를 넣어 미리 템플릿처럼 쉽게 사용이 가능하다. <br/>
예를 들면 아래와 같은 것도 가능하다.

아래 Component를 글 사이에 삽입하기

```javascript
import { useState } from 'react'

const TestComponent = () => {
  const [testText, setTestText] = useState<string>()

  const changeHandler = (e: React.FormEvent<HTMLInputElement>) => {
    const currentText = e.currentTarget.value
    setTestText(currentText)
  }

  return (
    <>
      <input
        type="text"
        placeholder="테스트 해주세요~~"
        onChange={changeHandler}
        className="block w-full mb-3 px-4 py-2 text-gray-900 bg-white border border-gray-300 rounded-md dark:border-gray-900 focus:ring-primary-500 focus:border-primary-500 dark:bg-gray-800 dark:text-gray-100"
      />
      당신이 작성한 것 : {testText}
    </>
  )
}

export default TestComponent
```

## SEO

블로그는 노출을 많이 되는 것이 중요하기에 내가 원하는 키워드와 유저 행동을 유도할 수 있다.
직접 만드는 블로그임으로 명시적으로 어떤 태그를 노출할 것인지 상세히 설정할 수 있어 공통 Component에서 next/head를 통해 손쉽게 SEO를 관리할 수 있다. 

추가로 마케팅 툴을 연결하는 작업도 쉽게 할 수 있었으며 대부분의 지표는 `GoogleSearchConsole`, `GoogleAnalaytics`를 사용해 확인하고 있다.

## 개발자 다운 (?) (커밋, 히스토리 관리)
코드로 뭐든 구현할 수 있습니다. 예약 포스팅, draft 관리 등 (sitemap에서 제외, 노출x)

커밋으로 관리하기 때문에 이력이 확실해서 좋으며 배포 관리도 할 수 있어 개발자스럽습니다. <br/>
(PS: NextJS 이미지 최적화 너무 좋습니다.)

## 확장성
위와 마찬가지지만 코드로 구현할 수 있고 플러그인, 라이브러리를 쉽게 넣을 수 있습니다.
NextJS apiServer를 활용하여 서버 역할도 할 수 있음으로 블로그의 끝판왕이라고 생각됩니다.

# 블로그 글 작성 톤앤매너 가이드

## 목적

이 블로그의 글은 AI와 협업하여 작성되는 경우가 많습니다. 문제는 AI 모델마다, 또 같은 모델이라도 세션마다 톤이 달라져서 블로그 전체의 일관성이 무너진다는 것입니다. 이 문서는 블로그 글 작성 시 어떤 톤과 스타일을 유지해야 하는지 정의합니다.

---

## 목표 톤: "경험을 나누는 동료 개발자의 존댓말"

기본 문체는 **"~했습니다 / ~입니다 / ~합니다"** 존댓말 기반의 편안한 문어체입니다.

격식 있는 존댓말이되, AI가 쓴 것 같은 딱딱한 교과서 톤은 피하고, 실제 경험을 공유하는 자연스러운 말투를 유지합니다.

---

## 현재 톤 문제 진단

### 1. 자연스러운 존댓말 (목표 톤)
- 특징: 1인칭 경험 서술, 솔직한 의견, "~했는데", "~더라고요", "~하더군요" 같은 자연스러운 연결
- 예: "실제로 써보니 생각보다 효과가 좋았습니다."
- 예: "처음에는 뭔가 했는데 알고 보니 이미 아는 내용이었습니다."

### 2. AI 교과서 톤 (피해야 할 톤)
- 특징: 독자를 가르치려는 뉘앙스, 과도한 격식, 매 문장이 완결된 명제처럼 들림
- 예: "이 글에서는 ~에 대해 깊이 있게 알아보겠습니다."
- 예: "문제는 대부분의 개발자가 ~한다는 것입니다. 이는 ~로 이어집니다."

### 3. AI 문학 과잉 톤 (피해야 할 톤)
- 특징: "위대한", "혁신적인", "기적을 보여주었습니다" 등 과한 수식어
- 예: "동시성과 패러다임의 위대한 타임라인"
- 예: "그들은 Virtual DOM이라는 무기를 통해 기적을 보여주었습니다."

## 기본 원칙

### 1. 문체는 존댓말 기반으로 통일합니다
- ✅ "실제로 써보니 꽤 괜찮았습니다."
- ✅ "왜 이게 필요한지는 당시 맥락을 보면 이해됩니다."
- ❌ "실제로 써보니 꽤 괜찮았다."
- ❌ "이번 포스팅에서는 ~에 대해 알아보겠습니다."

### 2. 1인칭 경험 중심으로 서술합니다
- ✅ "써보니까 이건 확실히 좋았습니다."
- ✅ "처음엔 의심스러웠는데, 직접 해보고 나니 달랐습니다."
- ❌ "개발자들은 이 기술을 활용하면 생산성을 높일 수 있습니다."

### 3. 과한 수식어와 미사여구를 쓰지 않습니다
- ✅ "꽤 유용합니다", "확실히 편합니다", "생각보다 괜찮았습니다"
- ❌ "혁신적인", "게임 체인저", "위대한", "획기적인", "눈부시게"

### 4. 기술 설명은 간결하게, 의견은 솔직하게 씁니다
- ✅ "Cookie는 4KB 제한이 있습니다. 서버에 자동 전송된다는 게 핵심입니다."
- ✅ "솔직히 이건 좀 오버 아닌가 싶었습니다."
- ❌ "Cookie의 핵심은 자동 전송입니다. fetch()나 XMLHttpRequest를 호출할 때 개발자가 명시적으로 추가하지 않아도 브라우저가 자동으로 Cookie를 HTTP 헤더에 포함시킵니다."

### 5. 독자를 '여러분'으로 부르지 않습니다
- ✅ (생략하거나) "혹시 이런 경험 있지 않으신가요?"
- ❌ "여러분도 이런 경험이 있으실 겁니다."
- ❌ "여러분의 프로젝트에서도 활용해보시기 바랍니다."

### 6. 단정형보다 의도형·제안형을 씁니다
- ✅ "기록해두려고 합니다." (의도 표현)
- ✅ "참고하시면 좋을 것 같습니다." (부드러운 제안)
- ❌ "기록해둡니다." (선언)
- ❌ "참고하시면 됩니다." (지시)

## 문장 패턴 예시

### 서론
```
# 좋은 예
최근에 XX를 쓰면서 이걸 정리해야겠다고 생각했습니다.
실제로 프로젝트에서 겪은 일인데, 공유하면 도움이 될 것 같아 기록합니다.

# 피해야 할 예
이번 포스팅에서는 XX에 대해 깊이 있게 알아보겠습니다.
XX는 현대 웹 개발에서 필수적인 기술입니다.
```

### 설명
```
# 좋은 예
간단히 말하면 XX는 ~하는 겁니다.
왜 이렇게 설계했냐면, 당시에는 Promise도 없던 시절이었기 때문입니다.

# 피해야 할 예
XX란 ~하는 기술을 의미합니다.
이는 현대 웹 애플리케이션의 복잡성이 증가함에 따라 필수적으로 요구되는 기술입니다.
```

### 결론
```
# 좋은 예
결국 상황에 맞게 고르면 됩니다. 정답은 없습니다.
이 글이 참고가 됐으면 합니다.

# 피해야 할 예
이 글이 여러분의 프로젝트에 도움이 되셨으면 합니다.
앞으로도 더 나은 개발 경험을 위해 함께 성장해 나가길 바랍니다.
```

## 피해야 할 표현 목록

| 피해야 할 표현 | 대체 표현 |
|---|---|
| ~에 대해 알아보겠습니다 | ~를 정리해봤습니다 |
| 여러분 | (생략하거나) 혹시 |
| 놀랍게도 | 의외로, 생각보다 |
| 혁신적인 / 획기적인 | 꽤 괜찮은, 확실히 다른 |
| ~해보시기 바랍니다 | ~해보시면 좋을 것 같습니다 |
| ~하시면 됩니다 | ~하시면 좋을 것 같습니다 |
| 필수적입니다 | 거의 필수입니다, 없으면 불편합니다 |
| 결정적 차이입니다 | 핵심 차이는 ~입니다 |
| 기존 브라우저 환경의 한계를 극복하는 | 기존에 안 되던 걸 가능하게 해주는 |
| 심도 있게 / 깊이 있게 | 제대로, 파고들어보면 |
| 도움이 되셨으면 좋겠습니다 | 참고가 됐으면 합니다 |
| 눈부시게 발전한 | 빠르게 발전한 |
| 위대한 / 거대한 | 중요한, 의미 있는 |
| 기록해둡니다 | 기록해두려고 합니다 |
| 정리합니다 | 정리해보려고 합니다 |
| ~의 산물 | ~의 결과, ~의 결과물 |
| ~의 산물이 바로 | ~로 탄생한 것이 바로 |

## 카테고리별 미세 조정

### 기술 글 (web, js, react, nextjs)
- 코드 예시 위주, 설명은 짧게
- "왜 이게 필요한가" → 개인 경험 연결
- 비교표/요약은 OK, 단 교과서 느낌 아닌 메모 느낌으로

### 회고/일상 (daily)
- 좀 더 자유롭게 1인칭 서술
- 감정 표현 자유롭게

### 여행 (travel)
- 실용 정보 중심
- 팁 형식: "~하면 됩니다", "~주의해야 합니다"

## AI에게 글 작성을 요청할 때 프롬프트 가이드

```
톤: "~했습니다 / ~입니다" 존댓말 기반, 1인칭 경험 중심
피해야 할 것: AI 교과서 톤, 과한 수식어, 독자를 가르치려는 뉘앙스
느낌: 옆에 있는 동료한테 "야 이거 해봤는데 이렇더라고" 하는 내용을 존댓말로 쓴 것
구조: 짧은 문장 위주, 코드 예시 중심, 의견은 솔직하게
```

## 기존 글 수정 가이드

기존에 AI 교과서 톤으로 작성된 글을 수정할 때:

1. "~입니다/합니다" 격식체는 유지하되 딱딱한 명제식 문장 완화
2. 서론의 "이 글에서는 ~를 다룹니다" → 왜 이 글을 쓰게 됐는지 경험 추가
3. "여러분" 호칭 제거
4. 과한 수식어(혁신적, 위대한, 눈부시게) → 담백한 표현으로 교체
   - 단순히 수식어만 지우면 문장이 어색해지는 경우가 있으므로, 뒤 문장까지 자연스럽게 수정
   - 예: "이 위대한 재설계의 산물이 바로 Fiber입니다" → "이 재설계로 탄생한 것이 바로 Fiber입니다"
5. "산물" 같은 AI 문어체 단어 → "결과물", "결과"로 교체
6. 결론의 "도움이 되셨으면" → 개인적 감상이나 실용적 요약으로 마무리
